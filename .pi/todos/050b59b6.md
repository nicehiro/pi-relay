## Description

Replace `SessionManager.inMemory()` with file-backed `SessionManager` in child sessions, using per-thread session directories so sessions survive master restarts.

## Current Behavior

- `SessionChild` uses `SessionManager.inMemory()` — all history lost on restart
- Thread-to-session mapping is purely in-memory (`children` Map in `index.ts`)
- Master restart kills all children with no way to resume

## Target Behavior

- Each child session persists to `~/.pi/agent/pi-relay/sessions/<threadId>/`
- On master restart, scan active (non-archived) threads in configured channels via Discord API
- For each thread with an existing session directory, resume via `SessionManager.continueRecent()`
- Send a "🔄 Session resumed" message to reconnected threads
- Stale session directories (thread archived/deleted while master was down) are kept on disk but not loaded

## Key SDK Findings

- **No custom session IDs** — `SessionManager.newSession()` always uses `randomUUID()` internally. The original assumption about `newSession({ sessionId })` was wrong.
- **File-backed sessions**: `SessionManager.create(cwd, sessionDir?)` for new, `SessionManager.continueRecent(cwd, sessionDir?)` to resume
- **cwd stored in session header** — `SessionManager.open(path)` extracts `cwd` from the header, so no separate manifest file is needed
- **Session directory convention**: `<sessionDir>/<timestamp>_<uuid>.jsonl`

## Approach

Use the thread ID as a **directory name** rather than a session ID:

1. **On spawn**: `SessionManager.create(cwd, '~/.pi/agent/pi-relay/sessions/<threadId>/')`
2. **On restart**: for each active thread, check if `~/.pi/agent/pi-relay/sessions/<threadId>/` exists → `SessionManager.continueRecent(cwd, sessionDir)` where `cwd` is read from the session header
3. **No manifest needed** — thread ID is the directory name, `cwd` is in the session header

## Implementation Notes

### SessionChild changes (`src/session-child.ts`)
- Accept a `sessionManager` parameter (instead of always creating `inMemory()`)
- New static factory: `SessionChild.create(threadId, discord, cwd)` — creates with `SessionManager.create(cwd, sessionDir)`
- New static factory: `SessionChild.resume(threadId, discord, sessionDir)` — resumes with `SessionManager.continueRecent()`, reads `cwd` from session header

### Reconnection logic (`src/index.ts`)
- On `session_start`, after Discord connects: scan active threads in configured channels
- For each thread, check if `~/.pi/agent/pi-relay/sessions/<threadId>/` exists
- If yes, create a `SessionChild` via the resume path and register it in the `children` Map
- Send "🔄 Session resumed" to each reconnected thread

### Reading cwd from existing sessions
- Use `findMostRecentSession(sessionDir)` (exported from SDK) to get the latest session file
- Then read the header to extract `cwd`
- Or use `SessionManager.continueRecent()` with a placeholder cwd then read `getCwd()` — need to verify which approach is cleaner

### Thread scanning
- Discord.js: `channel.threads.fetchActive()` on each configured channel
- Filter to threads that have a matching session directory

### Edge cases
- Thread archived/deleted while master was down → session dir exists but thread not in active list → skip, leave files on disk
- Session dir exists but session file is corrupted → `SessionManager` handles this (truncates and starts fresh per existing SDK code)
- Thread is active but session dir doesn't exist → not a pi-relay thread, ignore
