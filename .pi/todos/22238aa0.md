## Architecture

Router pattern: the main pi instance runs the relay extension and acts as a message router.

- **Main channel** → routes to the router pi itself (default session, current behavior)
- **Discord threads** → routes to spawned pi instances via their control sockets
- **One active thread at a time** — previous becomes inactive, only current one receives messages
- **Switching**: user sends message in old thread → warning + confirmation → switch active

## Decisions

- **`/new`**: creates a Discord thread (auto-named from first message) + spawns `pi` in a new tmux window
- **`/` commands**: all `/`-prefixed Discord messages forwarded to the target pi instance as commands
- **Spawned pi config**: uses defaults (cwd, model, thinking level). User describes what they want in their first prompt after `/new`
- **Restart**: start fresh, no reconnection to old threads/sessions
- **`/new` inside a thread**: creates a new thread in the channel (same as from main channel)

## Communication

Use pi's control socket to communicate with spawned instances:
- `list_sessions` → find spawned pi sessions
- `send_to_session` → forward Discord messages, wait for response
- Need to investigate: how to programmatically use control socket from extension code (Node.js `net` module to Unix socket, or `pi.exec()`)
